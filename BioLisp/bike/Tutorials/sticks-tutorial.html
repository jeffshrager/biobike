What is a Formal Language?


You are sitting in your isolated Antarctic work station monitoring the size of the atmospheric ozone hole. Truth be told, this doesn't occupy a whole lot of your attention, and you are without much enthusiasm twiddling the dial on your short-wave radio, hoping to pick up something to get you to the next hour. You're about to settle into a Chilean weather report when the radio suddenly stops working -- just static.

Then through the clicking and squealing you hear quite distinctly:

"Greetings, carbon-based creature!"

The surprising part is that the voice is a dead-on match for Mr. Rogers.

First contact! You grab your microphone. What can you say now, at perhaps THE critical juncture in human history? Amazingly, you can still think logically, and you consider that if "carbon-based" is worthy of alien comment, then ....

"Greetings," you respond, "non-carbon-based creature!"

"Very good!" chuckles Mr. Rogers.

Now what? Should you try to reach the Strategic Air Command to warn about flying saucers soon to hover over Washington? You decide to take instead the historic path towards friendly encounter.

"How is it going on Alpha Centauri?" A wild guess, you admit, but who knows?

"I am sorry," the alien responds. "Please provide the missing antecedent of the impersonal pronoun you used and I will be able to describe its mechanism of locomotion."

Hmmmmm. This is going to be more difficult than you thought.

"Never mind... How are things on your end?"

"Anterior or posterior?"

"...???."

Commentary

As you can see, English may not be the language of choice for a first encounter with ETs. You may consider this an insight of little practical value, but no, we face a similar problem all the time, even now: How do you communicate with computers?

Note that this is a more difficult problem than the one you faced in the Antarctic. The alien, one would suspect, accepted that you were conscious being and could use that fact to try to figure out what you mean. Computers - at least at present - have no internal model for your mind, and without one, understanding natural human languages is impossible.

And yet, we must communicate with computers, because we need them to do what we can't do by ourselves: direct complex manufacturing processes, coordinate sprawling communication systems, analyze huge sets of biological data. You might respond, "Why do I have to communicate with a computer? I don't communicate with my toaster.. I just put in the bread and push the button. " That would be fine if we could build a machine for every task at hand. But there are too many tasks, tasks whose processes must be continuously modified. Instead we have built a general machine - the digital computer - and are left with the problem of how to tell it what we want it to do.

We need a well-defined, formal language.

 

To some extent, English is a formal language. It has forms. For example:

It is also recursive - forms can lie within forms - leading to the possibility of endless variation. For example:

Nonetheless, English is a mess. To talk to a computer, we need something much more regular and unambiguous. Let's start small, building a language that ought to provide the best chance of allowing humans to speak to computers. It will have the following properties:

  1. It will be English-like (to accommodate us)
  2. It will have strict syntax (so the computer knows exactly where to find the subject, where to find the verb)
  3. It will be allow recursive forms (so we can express thoughts of some complexity)
  4. It will not require of the computer any background knowledge outside of simple addition.

From this groundwork, we'll build a language through which we can talk arithmetic to the computer. If we can't do even that, we're doomed.

Let's do it.

I. Preliminaries: Things and Functions

The language we'll build will have two possible forms:

  • FORM TYPE 1: Thing
  • FORM TYPE 2: Function

    • FORM TYPE 1: Thing(What did I just say?)

    1. Type a thing into the Command Box (the thin white window) and BioBIKE will tell you what you just typed in. Of course, the thing has to be a valid thing, according to the rules of the language, otherwise you'll get an error.

      2. Try typing anything that pops into your head (and press Enter). If you've got writer's block, they try this: 

      >>I am Sam
    2. Unless you're incredibly lucky (or you've played this game before), you got a response something like:

      Unknown asdfasdf

      3. That's because you didn't type in a valid THING. OK - we can't avoid it -- what is a thing? According to the rules of the language, a thing may consist of any of the following:

      • Quoted thing: Anything between double quotes [BioBIKE responds by repeating what you typed]
        • Try something like: 
        >>"I am Sam"
      • Asterisk thing: (an asterisk) -- [BioBIKE responds by repeating back its own last response]
        Try: 
        • >>*
      • Symbol thing: A symbol (or name) previously defined to be a valid thing [BioBIKE responds be telling you what the valid thing is] You'll learn in a moment how to define symbols, but a few symbols come automatically defined:

        • True or T, which stands for truth
        • False or NIL, which stands for falsity

        • Try this: 
        >>true
    3. I just introduced yet another concept - symbol. A symbol is any collection of characters (letters, numbers, etc) that does not contain a space or parenthesis. There are other restrictions, but that's enough for now.
      4. Try this: 
      >>Multiword symbol?
      and then this: 
      >>Multiword-symbol?

      The error message will tell you what set of characters BioBIKE considers to constitute a single symbol.

       

    FORM TYPE 2: Function (Do what I just said)

      Functions are always of the form:

      They:

      1. Begin with an open parenthesis
      2. Continue with the name of a valid function
      3. Continue (optionally) with one or more things or a functions
      4. End with a close parenthesis

      What number and kinds of things or functions follow the name of a function depends on the nature of the function. A command that doesn't fit this form is not a valid function (of course that doesn't mean that everything that does fit the form is a valid function).

    1. Try this: 
      >>SPELL fish
    2. That didn't work. Since what you typed didn't begin with a parenthesis, BioBIKE decided it couldn't be a function. So it must be a thing. But SPELL is not a valid thing, so error.
      3. Try this: 
      >>(SPELL fish)
    3. Still doesn't work, but at least different messages! The first was:

      Warning: While compiling these undefined functions were referenced:
      SPELL.

      BioBIKE recognized that SPELL was the name of a function, because it is the first symbol after an opening parenthesis. But it is not a function BioBIKE understands. The second message was:

      Attempt to take the value of the unbound variable `FISH'

      4. This one you've seen before. FISH isn't a valid thing. And it certainly isn't a function.

    4. Fix the second problem by putting the word between double quotes, making it a valid thing.
      5. Try this: 
      >>(SPELL "fish")

      Now BioBIKE recognized "fish" to be a thing, but it complains that it never heard of the function named SPELL before.

    5. How can we teach BioBIKE what SPELL means? The only way is to define it in terms of things BioBIKE already understands. Fortunately, BioBIKE has many previously defined functions. For Example:
      6. Try this: 
      >>(SPLIT "fish")

      That's probably something like what you may have intended by (SPELL "fish")

    6. Now we can define SPELL in terms of SPLIT, using the following complex function:


      7.  

      Try this (put it in the big window, hereafter called the Program Box): 

      >>(DEFINE-FUNCTION Spell (word) 
>>  (SPLIT word)
>>  )
    7. Now try again: 
      >>(SPELL "fish")

      8. The newly defined function SPELL took the thing it was given (represented internally by the symbol word) and gave it to the function SPLIT it into individual letters. These individual letters were returned to SPELL, which returned them to you.

      Any function, whether it is automatically provided or you define it yourself, returns the value of the last form in the function and that value replaces the function itself. We'll see this in action momentarily.

II. The Basic Stick Language

Let's begin to define a language through which we can speak arithmetic to a computer. Our starting point is that the computer already knows one and only one predefined mathematical concept: ONE-MORE-THAN.

  1. Try this: 
    >>"|"
    Call the thing you got a stick. The outcome should not be a surprise.


  2. Now give that result to a predefined function. Try this: 
    >>(ONE-MORE-THAN *)

    (Recall that the asterisk supplies the previous result returned by BioBIKE). You've just written a valid function, because the form:

    1. Begins with (
    2. Continues with a legal symbol ONE-MORE-THAN that has previously been defined as a function (it was predefined)
    3. Continues with a thing (asterisk is a thing)
    4. Ends with )

  3. Press Eval, and press Eval again. Do you get the idea of what ONE-MORE-THAN does?


  4. You can get the same result by putting one function inside another (i.e. nesting them).Try this: 
    >>(ONE-MORE-THAN (ONE-MORE-THAN (ONE-MORE-THAN "|")))


  5. Our language consists now solely of the thing "|" and the function ONE-MORE-THAN. Even with only two elements, you can do a great deal. With patience, you could construct a thing consisting of, say, a hundred sticks. How would you do it? (Don't try it!)


III. Extended Stick Language - Arithmetic

Addition

Using the function "ONE-MORE-THAN", we can define a function that performs addition of two groups of sticks. Suppose we have two bundles of sticks:

We want to add these two bundles together, but the computer only knows how to add one stick at a time. 

>>(ONE-MORE-THAN (ONE-MORE-THAN (ONE-MORE-THAN (ONE-MORE-THAN "|||"))))

This works though it's rather tedious to type ONE-MORE-THAN so many times.
But you get the idea that

is broken down to:

The concept of addition is that each stick in the second bundle is added to the first bundle until all sticks are gone. Keeping this notion in mind, we will now try to define a function "ADD-STICKS".

First, we will take a stepwise approach to understand the logic of defining a function called ADD-STICKS.

  1. Let's call the first bundle with three sticks "bundle1", the second bundle with four sticks "bundle2". 
    2. >> (DEFINE bundle1 as "|||")

    >> (DEFINE bundle2 as "||||")
  2. We want to add each stick in bundle2 into bundle1.

    Each successive iteration, the result is incremented by one stick. How many times should we perform the ONE-MORE-THAN operation? Exactly the same number as the number of sticks in bundle2.


    3. This operation can be automated in a LOOP. In a loop, you set a statement once, and the task is carried out repeatedly until the condition is met.



    Let's try a simple loop. 

    >> (FOR-EACH stick in bundle2
>>   (DISPLAY-LINE stick))
    You see all of the sticks in bundle2 displayed.

    Now initialize the variable. You want the starting number to be the same as the number of sticks in bundle1. 
    >> (FOR-EACH stick IN bundle2
>>    INITIALIZE result = bundle1
>>    (DISPLAY-LINE (ONE-MORE-THAN result)))
    The above doesn't work the way you expected – it just gives 1 + bundle1 sticks several times… because the result isn't UPDATED. To update: 
    >>(FOR-EACH stick IN bundle2
>>    INITIALIZE result = bundle1
>>    (ASSIGN result = (ONE-MORE-THAN result))                      
>>    (DISPLAY-LINE result))
    This result shows that each turn in the loop returns the result that is one more than the previous result.
Now we are going to combine what we just did within a function.
>>(DEFINE-FUNCTION ADD-STICKS
>>   REQUIRED (bundle1 (AND) bundle2)
>>   INITIALIZE result = bundle1
>>   BODY
>>     (LOOP FOR stick in bundle2
>>         (DEFINE result AS (ONE-MORE-THAN result)))
>>   (RETURN-FROM ADD-STICKS result))

We just created a function called "ADD-STICKS". Try:

>> (ADD-STICKS "|||" AND "||||")
